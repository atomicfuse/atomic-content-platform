## Decisions

### Decision 1: Two-color model on top of variant-owned neutrals (vs. one color + auto-derive, vs. full palette)
**Alternatives considered:**
1. **One main color, rest auto-derived** — wizard becomes one click. Risk: visually flat sites; auto-derivation rarely matches brand intent.
2. **Two colors (primary header + accent CTA)** — matches what reference screenshots actually show; minimum viable brand identity.
3. **Full palette** — power-user friendly but decision-fatigue heavy in the wizard.

**Chosen:** Two colors. Stored as `theme.colors.primary` + `theme.colors.accent` (existing schema). Variant `theme.css` reads only these two and supplies its own neutrals.
**Trade-offs accepted:** Sites can't fully customise tertiary colors (`text`, `surface`, `border`) without dropping into yaml. Acceptable for v1; the existing `theme.colors` Record<string,string> still allows overrides for power users.

### Decision 2: Frontmatter flag `featured: hero|must-read` for curated slots (vs. auto-from-latest, vs. dashboard editor's-pick UI)
**Chosen:** Frontmatter flag with auto-fallback. Sites with no flagged articles still render correctly. Editing one yaml field beats building a drag-drop UI for v1.
**Trade-offs accepted:** Editors set the flag inline in markdown — no dashboard UI for v1. Backlog item to add a dashboard "Featured" column.

### Decision 3: Per-site `theme.layout_v2: true` toggle for safe rollout (vs. flip everyone at once)
**Chosen:** Toggle. Phase 2 builds new components against `coolnews-atl` only; Phase 4 flips remaining sites one at a time after visual sanity checks; Phase 4.2 removes the toggle once everyone is on the new layout.
**Trade-offs accepted:** Keeps both code paths in `index.astro` and `[slug]/index.astro` until Phase 4.2 — temporary code duplication. Worth it for the ability to roll back per-site without redeploy.

### Decision 4: Site-worker `/api/articles` endpoint for Load More (vs. routing through dashboard service)
**Chosen:** Endpoint lives on the site-worker. Reads CONFIG_KV directly via the same middleware that serves the rest of the site. Cache headers match the homepage (`s-maxage=60, stale-while-revalidate=300`).
**Trade-offs accepted:** Each Worker carries the endpoint code (cheap; ~50 LOC). Alternative would be a dashboard endpoint that the worker proxies — adds latency and a cross-service dependency for no real gain.

### Decision 5: Subagent-driven execution with phase checkpoints (vs. fully autonomous, vs. parallel session)
**Chosen:** Subagent-driven in this session, with a user checkpoint at each phase boundary (Phase 1 done; pause for review before Phase 2).
**Trade-offs accepted:** More wall-clock time (10–15 min/task × 21 tasks × 1–2 review iterations). Earned: TDD discipline per task, fresh subagent context (no pollution), every commit reviewed for spec compliance + quality before being marked done.

### Decision 6: Skip Task 1.8 (org.yaml defaults) until user authorizes
**Chosen:** Pause Phase 1 at Task 1.7. Task 1.8 pushes to `main` of the **network** repo, which triggers `sync-kv.yml` → re-seeds production CONFIG_KV. Even though the new fields are additive and the production Worker will ignore unknown keys, "push to main of a prod data repo without explicit confirmation" violates the system prompt's "executing actions with care" guidance.
**Trade-offs accepted:** Phase 1 is 7/8 complete on resume. The remaining task is a small yaml edit; trivial to finish in <5 minutes with confirmation.
