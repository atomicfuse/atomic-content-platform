# Sites: Creating, Editing, and Deploying

## Site Lifecycle

Every site moves through these statuses:

```
New --> Staging --> Preview --> Ready --> Live
```

| Status | Meaning |
|--------|---------|
| **New** | Domain exists in Cloudflare but no site config yet |
| **Staging** | Wizard completed. Site files committed to a staging branch, Cloudflare Pages project created, preview build running |
| **Preview** | Staging build is ready for review at a `*.pages.dev` URL |
| **Ready** | Staging merged to main. Production build deployed to `*.pages.dev`. No custom domain yet |
| **Live** | Custom domain attached and DNS verified |

There is also a **WordPress** status for legacy sites being migrated.

## Creating a Site (Wizard Flow)

The dashboard wizard at `/wizard` has five steps:

### Step 1: Identity
- **Site name** and **tagline**
- **Pages project name** (becomes the `*.pages.dev` subdomain)
- **Company** (ATL or NGC) and **Vertical** (Tech, Travel, etc.)

### Step 2: Content Brief
- **Audience** description, **tone**, **topics** (auto-suggested by Gemini)
- **Content guidelines** (free-form)
- **Schedule**: articles per week, preferred days

### Step 3: Theme
- **Base theme**: modern, editorial, bold, or classic
- **Logo**: auto-generated by Gemini with background removal, or upload custom

### Step 4: Preview
- Review all settings before committing

### Step 5: Go Live
- Creates the Cloudflare Pages project
- Commits `site.yaml`, `skill.md`, and asset directories to a `staging/{project}` branch in the data repo
- Triggers a GitHub Actions build
- Shows the staging preview URL

## What Gets Created

When the wizard completes, the following files are committed to the data repo:

```
sites/{project-name}/
  site.yaml          -- full site configuration
  skill.md           -- AI agent instructions derived from the brief
  assets/
    logo.png         -- generated or uploaded logo (optional)
  articles/
    .gitkeep
```

And in the data repo root:

```
dashboard-index.yaml -- updated with the new site entry
```

## site.yaml Structure

```yaml
domain: my-travel-site
site_name: "Wanderlust Weekly"
site_tagline: "Your guide to the world"
pages_project: my-travel-site
groups:
  - travel          # editorial vertical
  - premium-ads     # ad partner config
active: true
brief:
  audience: "Adventure travelers aged 25-45"
  tone: "conversational"
  article_types:
    listicle: 40
    standard: 30
    how-to: 20
    review: 10
  topics:
    - Destination Guides
    - Travel Tips
    - Local Culture
  seo_keywords_focus: []
  content_guidelines:
    - "Focus on practical advice"
    - "Include budget estimates"
  review_percentage: 5
  schedule:
    articles_per_week: 5
    preferred_days: ["Monday", "Wednesday", "Friday"]
    preferred_time: "10:00"
  vertical: Travel
theme:
  base: modern
  logo: /assets/logo.png
  favicon: /assets/logo.png
```

## Staging Branch Workflow

Each site gets a dedicated staging branch: `staging/{project-name}`.

1. **All edits go to staging first** -- config changes, new articles, logo updates
2. **Preview URL** -- Cloudflare Pages builds from the staging branch and provides a preview at `https://staging-{project}.{subdomain}.pages.dev`
3. **Go Live / Publish** -- merges staging to `main`, which triggers the production build
4. **Branch reset** -- after merge, staging is deleted and recreated from `main` HEAD so it is clean for the next edit cycle

## Editing an Existing Site

From the site detail page (`/sites/{domain}`), operators can:

- **Edit config** (name, tagline, brief, theme) -- updates `site.yaml` on the staging branch
- **Generate/upload logo** -- commits to `assets/logo.png` on staging
- **Delete articles** -- removes markdown files from staging
- **Publish staging** -- merges staging to main for production

All edits trigger a rebuild on the staging branch for preview before publishing.

## Attaching a Custom Domain

1. Site must be in **Ready** status (staging merged, production build live)
2. From the site detail page, enter the custom domain
3. Dashboard calls Cloudflare API to add the domain to the Pages project
4. Set up DNS: CNAME record pointing to `{project}.pages.dev`
5. Status updates to **Live** once DNS is verified

## Deleting a Site

Full cleanup in order:
1. Delete staging branch from Git
2. Delete all site files from the data repo (`sites/{domain}/`)
3. Delete the Cloudflare Pages project
4. Move the entry to trash in `dashboard-index.yaml`

Sites in trash can be restored or permanently deleted.
